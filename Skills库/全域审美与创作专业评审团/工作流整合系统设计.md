## 二、核心组件设计

### 1. 主编排器（Master Orchestrator）

这是整个系统的入口，负责：
- 接收用户请求
- 分析需求类型
- 路由到对应的Agent或直接执行
- 整合跨文件夹的资源

### 2. 节点注册系统

将你现有的功能节点注册到统一的系统中：

```json
// nodes/node-registry.json
{
  "version": "1.0",
  "nodes": [
    {
      "id": "NODE-001",
      "name": "内容评审节点",
      "type": "agent-based",
      "location": "skills/全域审美与创作专业评审团.md",
      "agent": "content-review-agent",
      "capabilities": ["评审", "打分", "优化建议"],
      "input": {
        "required": ["content", "type"],
        "optional": ["audience", "platform"]
      },
      "output": {
        "format": "review-report",
        "fields": ["score", "grade", "suggestions"]
      }
    },
    {
      "id": "NODE-002",
      "name": "[你的现有节点1]",
      "type": "function-based",
      "location": "nodes/[文件夹名]/[文件名]",
      "capabilities": ["..."],
      "input": {...},
      "output": {...}
    }
    // ... 更多节点
  ]
}
```

### 3. 跨文件夹工作流定义

```yaml
# integrations/cross-folder-workflows.yaml

workflows:
  - id: "WF-001"
    name: "完整内容创作流程"
    description: "从创意到发布的完整流程"
    
    nodes:
      - step: 1
        node_id: "NODE-XXX"  # 你现有的创意生成节点
        location: "nodes/creative-generation/"
        output_var: "creative_idea"
      
      - step: 2
        node_id: "NODE-001"  # 新的评审节点
        location: "skills/全域审美与创作专业评审团.md"
        input_from: "creative_idea"
        output_var: "review_result"
      
      - step: 3
        node_id: "NODE-XXX"  # 你现有的优化节点
        location: "nodes/content-optimization/"
        input_from: ["creative_idea", "review_result"]
        output_var: "optimized_content"
      
      - step: 4
        node_id: "NODE-XXX"  # 你现有的发布节点
        location: "nodes/publishing/"
        input_from: "optimized_content"
        output_var: "publish_result"
    
    output:
      - "optimized_content"
      - "review_result"
      - "publish_result"
```

## 三、整合实施方案

### 方案A：渐进式整合（推荐）

#### 阶段1：盘点现有资源（1天）

1. **列出所有现有节点**
```bash
# 创建节点清单
你的现有节点：
- 节点1：[名称] - [位置] - [功能]
- 节点2：[名称] - [位置] - [功能]
- ...
```

2. **分析节点类型**
```
类型A：纯函数型（输入→处理→输出）
类型B：需要AI推理型（需要SubAgent）
类型C：工具调用型（需要特定工具权限）
```

3. **识别依赖关系**
```
节点A → 节点B → 节点C
     ↘ 节点D ↗
```

#### 阶段2：创建统一注册表（1天）

创建三个核心注册文件：

**1. 节点注册表**
```json
// nodes/node-registry.json
{
  "nodes": [
    {
      "id": "NODE-001",
      "name": "内容评审",
      "path": "skills/全域审美与创作专业评审团.md",
      "type": "agent-based",
      "agent": "content-review-agent"
    },
    {
      "id": "NODE-002", 
      "name": "[你的节点]",
      "path": "nodes/[路径]",
      "type": "function-based"
    }
  ]
}
```

**2. Agent注册表**
```json
// agents/agent-registry.json
{
  "agents": [
    {
      "id": "content-review-agent",
      "name": "ContentReviewAgent",
      "skill_path": "skills/全域审美与创作专业评审团.md",
      "tools": ["readFile", "fsWrite"],
      "nodes": ["NODE-001"]
    }
  ]
}
```

**3. 工作流注册表**
```json
// workflows/workflow-registry.json
{
  "workflows": [
    {
      "id": "WF-001",
      "name": "内容创作全流程",
      "nodes": ["NODE-XXX", "NODE-001", "NODE-XXX"],
      "definition": "workflows/content-creation.yaml"
    }
  ]
}
```

#### 阶段3：创建主编排器（2天）

这是系统的核心入口，负责统一调度。

#### 阶段4：逐步迁移（按需）

不需要一次性迁移所有节点，可以：
1. 先整合最常用的工作流
2. 保留现有节点的独立使用方式
3. 逐步将节点接入统一系统

## 四、主编排器设计

让我创建一个完整的主编排器：

```markdown
# 主编排器 (Master Orchestrator)

## 系统定位
你是整个工作流系统的主编排器，负责：
1. 接收用户请求
2. 识别需要调用的节点/Agent
3. 协调跨文件夹的资源
4. 执行完整工作流
5. 整合并返回结果

## 核心能力

### 1. 资源发现
能够自动发现和加载：
- skills/ 目录下的所有Skill
- agents/ 目录下的所有Agent定义
- nodes/ 目录下的所有功能节点
- workflows/ 目录下的所有工作流定义

### 2. 智能路由
根据用户需求，自动判断：
- 是否需要调用SubAgent
- 需要哪些节点参与
- 执行顺序（串行/并行）
- 数据如何在节点间传递

### 3. 工作流执行
支持多种执行模式：
- 单节点执行
- 预定义工作流
- 动态组合工作流
- 条件分支
- 循环迭代

## 使用方式

### 方式1：直接调用节点
\`\`\`
用户：评审我的短视频脚本

主编排器：
1. 识别需求：内容评审
2. 查找节点：NODE-001（内容评审节点）
3. 检查类型：agent-based
4. 调用SubAgent：content-review-agent
5. 返回结果
\`\`\`

### 方式2：执行预定义工作流
\`\`\`
用户：执行完整的内容创作流程

主编排器：
1. 识别工作流：WF-001（内容创作全流程）
2. 加载工作流定义
3. 按顺序执行各节点
4. 传递中间结果
5. 返回最终输出
\`\`\`

### 方式3：动态组合
\`\`\`
用户：先生成创意，然后评审，如果不达标就优化，再评审

主编排器：
1. 分析需求：创意生成 → 评审 → 条件判断 → 优化 → 再评审
2. 动态构建工作流
3. 执行并监控
4. 返回结果
\`\`\`

## 命令系统

### 查询命令
- `@编排器 列出所有节点` - 显示所有可用节点
- `@编排器 列出所有工作流` - 显示所有预定义工作流
- `@编排器 查看节点 NODE-001` - 查看节点详情
- `@编排器 查看工作流 WF-001` - 查看工作流定义

### 执行命令
- `@编排器 执行节点 NODE-001` - 执行单个节点
- `@编排器 执行工作流 WF-001` - 执行预定义工作流
- `@编排器 创建工作流 [定义]` - 动态创建工作流

### 管理命令
- `@编排器 注册节点 [配置]` - 注册新节点
- `@编排器 注册工作流 [配置]` - 注册新工作流
- `@编排器 更新注册表` - 重新扫描并更新注册表
```

## 六、整合示例

### 示例：整合你的现有"创意生成"节点

假设你有一个创意生成节点在 `nodes/creative-gen/generator.md`

**步骤1：注册节点**
```json
{
  "id": "NODE-CREATIVE-001",
  "name": "创意生成器",
  "path": "nodes/creative-gen/generator.md",
  "type": "agent-based",
  "capabilities": ["创意生成", "头脑风暴"],
  "input": {
    "required": ["topic", "style"],
    "optional": ["constraints"]
  },
  "output": {
    "format": "creative-idea",
    "fields": ["title", "concept", "key_points"]
  }
}
```

**步骤2：创建工作流**
```yaml
# workflows/creative-to-publish.yaml
name: "从创意到发布"
steps:
  - node: "NODE-CREATIVE-001"  # 你的创意生成
    output: "idea"
  
  - node: "NODE-001"            # 新的评审节点
    input: "${idea}"
    output: "review"
  
  - node: "NODE-OPTIMIZE-001"   # 你的优化节点
    input: ["${idea}", "${review}"]
    output: "final"
```

**步骤3：使用**
```
用户：@编排器 执行工作流 creative-to-publish
主题：美食探店
风格：轻松幽默
```

## 七、常见问题解答

### Q1：我的现有节点不需要SubAgent，怎么办？
A：没问题！节点类型可以是：
- `agent-based`：需要SubAgent执行
- `function-based`：直接函数调用
- `prompt-based`：简单的Prompt模板
- `tool-based`：调用特定工具

### Q2：不同文件夹的节点如何通信？
A：通过标准化的数据格式：
```json
{
  "from_node": "NODE-001",
  "to_node": "NODE-002",
  "data": {
    "content": "...",
    "metadata": {...}
  }
}
```

### Q3：可以保留原有的使用方式吗？
A：完全可以！整合后：
- 原有节点仍可独立使用
- 新增了通过编排器调用的方式
- 两种方式并存，互不影响

### Q4：如何处理节点版本更新？
A：在注册表中记录版本：
```json
{
  "id": "NODE-001",
  "version": "4.0",
  "updated": "2024-01-15"
}
```
